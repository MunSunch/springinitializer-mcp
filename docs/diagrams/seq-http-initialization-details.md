# Инициализация HTTP-сессии — подробное описание шагов

Подробное описание каждого нумерованного шага из `seq-http-initialization.puml`.

## Шаг 1 — Клиент отправляет запрос initialize
MCP-клиент отправляет HTTP-запрос `POST /mcp` с JSON-RPC телом `initialize`. Заголовок `Mcp-Session-Id` ещё отсутствует. В заголовках должен быть `Accept: application/json, text/event-stream`.

## Шаг 2 — Express проверяет условия запроса
Обработчик маршрута Express проверяет: заголовок сессии отсутствует И метод в теле запроса — `initialize`. Это идентифицирует запрос как создание новой сессии.

## Шаг 3 — Создание StreamableHTTPServerTransport
Создаётся новый экземпляр `StreamableHTTPServerTransport` с генератором UUID для идентификаторов сессий. Каждая сессия получает собственный экземпляр транспорта.

## Шаг 4 — Создание экземпляра McpServer
Создаётся новый `McpServer` для данной сессии. Каждая HTTP-сессия получает независимый экземпляр сервера для изоляции состояния.

## Шаг 5 — Регистрация инструментов, ресурсов, промптов
На новом экземпляре сервера регистрируются все MCP-возможности: инструмент генерации проекта, ресурсы с метаданными и шаблон промпта.

## Шаг 6 — Подключение сервера к транспорту
`server.connect(transport)` привязывает обработчик MCP-протокола к HTTP-транспорту данной сессии.

## Шаг 7 — Обработка запроса initialize
Транспорт обрабатывает JSON-RPC сообщение `initialize` из тела запроса.

## Шаг 8 — Маршрутизация initialize к серверу
Транспорт передаёт метод внутреннему обработчику MCP-сервера.

## Шаг 9 — Сервер формирует ответ
Сервер отвечает со своими возможностями, версией протокола и информацией о сервере.

## Шаг 10 — Транспорт формирует SSE-ответ
Ответ оборачивается в Server-Sent Event (SSE) с заголовком `Mcp-Session-Id`.

## Шаг 11 — Сохранение сессии в Map
Сервер сохраняет связку `sessionId -> transport` для маршрутизации последующих запросов от этого клиента к правильному транспорту.

## Шаг 12 — Ответ отправлен клиенту
HTTP-ответ 200 с `Content-Type: text/event-stream` и заголовком `Mcp-Session-Id`. Клиент должен включать этот заголовок во все последующие запросы.

## Шаг 13 — Клиент отправляет tools/list
Клиент делает ещё один запрос `POST /mcp`, на этот раз с заголовком `Mcp-Session-Id` из предыдущего ответа.

## Шаг 14 — Поиск сессии
Express ищет идентификатор сессии в Map и находит существующий транспорт.

## Шаг 15 — Обработка запроса
Существующий транспорт обрабатывает запрос `tools/list`.

## Шаг 16 — Маршрутизация к серверу
Транспорт направляет метод MCP-серверу.

## Шаг 17 — Сервер возвращает список инструментов
Сервер возвращает список зарегистрированных инструментов (`generate-spring-boot-project`) с их схемами.

## Шаг 18 — Формирование SSE-ответа
Ответ форматируется как SSE.

## Шаг 19 — Ответ отправлен
Клиент получает список инструментов и может их вызывать.
